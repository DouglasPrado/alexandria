# ADR-001: NestJS (TypeScript) como stack principal do backend

**Data:** 2026-03-16

**Status:** Aceita

**Autores:** Douglas Prado

---

## Contexto

O Alexandria é um sistema de armazenamento distribuído que executa operações IO-bound pesadas: chunking de arquivos (~4MB), criptografia AES-256-GCM, hashing SHA-256, replicação de chunks entre nós, e transcodificação de mídia. O sistema precisa rodar como orquestrador em VPS e como agente de nó em containers Docker. Time de 1 pessoa mantém todo o stack.

---

## Drivers de Decisão

- Produtividade: time de 1 pessoa precisa de framework opinado com convenções claras
- Type-safety: TypeScript reduz bugs em runtime com verificação estática de tipos
- Concorrência: heartbeats, replicação e pipeline de mídia rodam em paralelo via event loop
- Ecossistema: NPM oferece o maior ecossistema de packages do mercado
- Manutenibilidade de longo prazo: sistema deve funcionar por décadas

---

## Opções Consideradas

### Opção A: NestJS (TypeScript)

Framework Node.js opinado com decorators, dependency injection e arquitetura modular.

| Aspecto | Avaliação |
|---------|-----------|
| Prós | Type-safety nativa do TypeScript; dependency injection built-in; arquitetura modular; ecossistema maduro (@nestjs/core, prisma, @nestjs/schedule); mesmo runtime para frontend e backend |
| Contras | Overhead do framework em projetos pequenos; performance inferior a linguagens compiladas para CPU-bound |
| Esforço | Baixo |
| Risco | Baixo |

### Opção B: Express (TypeScript)

Framework minimalista e flexível para Node.js.

| Aspecto | Avaliação |
|---------|-----------|
| Prós | Extremamente flexível; comunidade enorme; middleware simples; baixa curva de aprendizado |
| Contras | Sem opinião arquitetural — requer mais decisões manuais; sem DI nativo; sem estrutura modular |
| Esforço | Baixo |
| Risco | Médio (decisões arquiteturais ad-hoc) |

### Opção C: Fastify (TypeScript)

Framework web performático para Node.js com schema-based validation.

| Aspecto | Avaliação |
|---------|-----------|
| Prós | Performance superior ao Express; JSON Schema validation; plugin system; TypeScript support |
| Contras | Ecossistema menor que Express; sem DI nativo; menos opinado que NestJS |
| Esforço | Baixo |
| Risco | Baixo |

---

## Decisão

**Escolhemos NestJS (TypeScript)** porque o framework oferece arquitetura modular com dependency injection, decorators e convenções claras — ideal para um time de 1 pessoa que precisa manter o sistema por décadas. O TypeScript garante type-safety nativa e a integração com Prisma oferece queries type-safe. Operações de criptografia utilizam crypto nativo Node.js e @noble/ciphers. O deploy via container Docker simplifica a distribuição para orquestrador e agentes de nó.

---

## Consequências

### Positivas

- Arquitetura modular e opinada — reduz decisões ad-hoc e facilita manutenção
- Type-safety nativa do TypeScript — reduz bugs em runtime
- Container Docker para orquestrador e agentes de nó — deploy consistente em qualquer ambiente
- Ecossistema NPM extenso — packages para praticamente qualquer necessidade
- Mesmo runtime (Node.js) para frontend e backend — reduz context switching

### Negativas

- Performance inferior a linguagens compiladas para operações CPU-bound
- Overhead do framework para projetos muito simples
- Node.js event loop é single-threaded — operações CPU-bound pesadas precisam de worker threads

### Riscos

- Performance em operações crypto/hashing — **Mitigação:** usar crypto nativo Node.js (C++ bindings) e worker threads para operações CPU-bound
