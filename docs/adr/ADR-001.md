# ADR-001: Rust como linguagem principal do backend

**Data:** 2026-03-16

**Status:** Aceita

**Autores:** Douglas Prado

---

## Contexto

O Alexandria é um sistema de armazenamento distribuído que executa operações IO-bound pesadas: chunking de arquivos (~4MB), criptografia AES-256-GCM, hashing SHA-256, replicação de chunks entre nós, e transcodificação de mídia. O sistema precisa rodar como orquestrador em VPS e como agente de nó em dispositivos heterogêneos (PCs, NAS, VPS). Time de 1 pessoa mantém todo o stack.

---

## Drivers de Decisão

- Performance: operações de criptografia e IO dominam o workload
- Binário único: agentes de nó precisam rodar sem runtime/dependências
- Memory safety: bugs de memória em sistema de storage = corrupção de dados
- Concorrência: heartbeats, replicação e pipeline de mídia rodam em paralelo
- Manutenibilidade de longo prazo: sistema deve funcionar por décadas

---

## Opções Consideradas

### Opção A: Rust

Linguagem de sistemas com zero-cost abstractions, memory safety garantida em compile-time, sem GC.

| Aspecto | Avaliação |
|---------|-----------|
| Prós | Performance nativa; memory safety sem GC; binário estático; async maduro (Tokio); ecossistema crypto robusto (ring, aes-gcm) |
| Contras | Curva de aprendizado íngreme; compile times mais longos; ecossistema menor que Go para tooling de infra |
| Esforço | Alto (aprendizado inicial) |
| Risco | Médio (produtividade mais lenta no início) |

### Opção B: Go

Linguagem de sistemas com GC, concorrência via goroutines, compilação rápida.

| Aspecto | Avaliação |
|---------|-----------|
| Prós | Compilação rápida; goroutines simples; binário único; ecossistema forte para infra/cloud |
| Contras | GC pode causar latency spikes em IO pesado; menos controle de memória; generics limitados |
| Esforço | Médio |
| Risco | Baixo |

### Opção C: Node.js (TypeScript)

Runtime JavaScript com event loop, ecossistema NPM extenso.

| Aspecto | Avaliação |
|---------|-----------|
| Prós | Produtividade alta; mesmo runtime para frontend e backend; ecossistema enorme |
| Contras | Single-threaded; performance insuficiente para crypto/hashing; overhead de runtime em agentes de nó; V8 GC |
| Esforço | Baixo |
| Risco | Alto (performance, agentes de nó) |

---

## Decisão

**Escolhemos Rust** porque o workload é dominado por operações que se beneficiam de performance nativa e controle de memória: criptografia AES-256-GCM (~GB/s nativo vs ~100MB/s em Node), hashing SHA-256 de chunks, IO paralelo para replicação. O binário estático sem runtime é ideal para agentes de nó em dispositivos variados. O custo de aprendizado é compensado pela durabilidade do sistema (décadas).

---

## Consequências

### Positivas

- Performance nativa para crypto, hashing e IO — sem overhead de GC ou runtime
- Binário único para agentes de nó — sem instalar Node/Go/Python no dispositivo
- Memory safety em compile-time — reduz risco de corrupção de dados por bugs de memória
- Ecossistema async maduro (Tokio) para concorrência pesada

### Negativas

- Curva de aprendizado mais íngreme que Go
- Compile times mais longos (~2-5min para projeto completo)
- Ecossistema menor para tooling de infra/cloud vs Go

### Riscos

- Produtividade inicial mais lenta — **Mitigação:** usar crates battle-tested (axum, sqlx, tokio) em vez de reinventar
