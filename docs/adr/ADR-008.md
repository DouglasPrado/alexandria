# ADR-008: Orquestrador como apps/api em vez de packages/orchestrator

**Data:** 2026-03-25

**Status:** Aceita

**Autores:** Douglas Prado

---

## Contexto

O build plan (docs/blueprint/11-build_plan.md) define o workspace com 3 packages: `core-sdk`, `orchestrator` e `node-agent`. Durante a implementação da Fase 0, o orquestrador foi criado como `apps/api` (uma aplicação NestJS completa) em vez de `packages/orchestrator` (um package de biblioteca).

A distinção é relevante porque `apps/` contém aplicações deployáveis com seu próprio runtime, enquanto `packages/` contém bibliotecas reutilizáveis sem runtime próprio. O orquestrador é um servidor NestJS com API REST, Swagger, guards, middlewares e lifecycle hooks — comportamento de aplicação, não de biblioteca.

---

## Drivers de Decisão

- Clareza semântica do monorepo (apps vs packages)
- Convenção do ecossistema pnpm workspaces / Turborepo
- Simplicidade operacional (princípio arquitetural #6)

---

## Opções Consideradas

### Opção A: packages/orchestrator (conforme build plan original)

Orquestrador como package em `packages/orchestrator/`.

| Aspecto | Avaliação |
| ------- | --------- |
| Prós    | Consistência com build plan original |
| Contras | Semanticamente incorreto — NestJS app não é uma biblioteca; confunde papel de apps vs packages no monorepo |
| Esforço | Baixo |
| Risco   | Baixo |

### Opção B: apps/api (implementação atual)

Orquestrador como aplicação em `apps/api/` com nome `@alexandria/api`.

| Aspecto | Avaliação |
| ------- | --------- |
| Prós    | Semântica correta (app deployável); convenção do ecossistema; Docker build aponta para apps/; scripts de dev/build/start fazem sentido em apps/ |
| Contras | Diverge do build plan original |
| Esforço | Baixo |
| Risco   | Baixo |

---

## Decisão

**Escolhemos a Opção B: apps/api** porque o orquestrador é uma aplicação NestJS deployável com runtime próprio, não uma biblioteca. A convenção `apps/` para aplicações e `packages/` para bibliotecas é padrão no ecossistema pnpm/Turborepo e melhora a clareza do monorepo.

Estrutura resultante:

```
apps/
  api/          ← Orquestrador (NestJS, API REST)
  web/          ← Web client (Next.js 16)
packages/
  core-sdk/     ← Biblioteca de crypto, chunking, hashing
  node-agent/   ← Agente de nó (futuro: pode migrar para apps/)
```

---

## Consequências

### Positivas

- Monorepo segue convenção padrão do ecossistema
- Docker build e CI referenciam `apps/api` corretamente
- Fica claro que core-sdk é dependência de apps/api (e não o inverso)

### Negativas

- Build plan original precisa ser interpretado com esta divergência em mente

### Riscos

- `node-agent` também é uma app deployável — pode precisar migrar para `apps/node-agent/` no futuro — **Mitigação:** avaliar na Fase 1 quando node-agent for implementado
